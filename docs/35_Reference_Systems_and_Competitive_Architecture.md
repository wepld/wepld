# 35 — Reference Systems and Competitive Architecture Study

**Status:** Proposed architecture evidence for review. It is not implementation authorization.

**Evidence cutoff:** 2026-07-14. Source repositories are pinned below to the observed commit. Documentation links are first-party pages as observed on the cutoff date and may later change.

## Purpose and decision boundary

This study tests reference-system ideas against WePLD's architecture; it does not turn their feature lists into a product backlog. The comparison asks which underlying primitives improve governed, evidence-backed engineering and which trust assumptions conflict with WePLD.

The required synthesis is:

- **GitHub Spec Kit informs the WePLD Delivery Protocol:** its staged flow becomes typed, versioned domain state and governed transitions rather than authoritative Markdown prompts.
- **Pi informs the composable Hermes harness:** a small agent/session kernel, dynamic context, skills, extensions, packages, compaction, SDK, RPC, and event projections are useful seams, but never authority boundaries.
- **Zed and ACP inform editor interoperability and review:** ACP is a future Core-mediated adapter and Zed is a possible H9 surface, not the foundation of Core.
- **Warp informs execution operations:** terminal-native task visibility, routing, permission presentation, orchestration, and budget views inform bounded Hermes scheduling and an Execution Console view inside Mission Control. WePLD is not a terminal product or cloud-first control plane.
- **WePLD supplies the invariant:** Core alone owns durable truth, admission, policy, capability issuance, approvals, effect intent/result, evidence acceptance, recovery truth, and completion. Mandatory governance does not become optional because an adapter or agent is trusted.

Draft PR #1 remains an open, unmerged, unratified reference. This study neither changes it nor treats it as canonical.

## Method and interpretation rules

1. Evidence is restricted to official documentation and official repositories. Product documentation establishes advertised behavior, not an unobserved implementation guarantee.
2. **Observed** means a first-party source states or exposes the capability. **Inference** means the WePLD conclusion derived from those observations. **Not publicly evidenced** means the reviewed first-party material did not establish the claim; it does not prove absence.
3. Each system is assessed against the same 16 dimensions. Source keys in the assessment and matrix resolve to the evidence ledger below.
4. The matrix records **candidate target dispositions**, not completed admission decisions. `Adopt` means the candidate target is an independently derived design principle; `Adapt` means the candidate target transforms the principle to preserve WePLD authority. Both remain `Candidate` for roadmap admission until their experiment, review, ADR, and milestone gate pass. `Experiment` withholds even a target design commitment. `Defer` excludes the idea from H1–H9 until its stated trigger. `Reject` records a conflicting principle. None means copy source, and each matrix row has exactly one disposition.
5. Every `Adopt`, `Adapt`, and `Experiment` row is tied to a preregistered controlled spike or ablation. Passing a spike supplies architecture evidence; it cannot authorize production implementation or bypass the applicable ADR and H1–H9 entry gate.
6. Worktrees are file-view isolation, not process, network, credential, or external-effect isolation. Transcripts are not the Core ledger. Summaries are disposable context projections. LSP/index results are observational evidence. Review recommendations are not completion authority.

## Uniform assessment dimensions

The numbered dimensions used in every system table are: **D01** core harness primitives; **D02** specification and planning workflow; **D03** context construction and compaction; **D04** skills and extension model; **D05** tool interfaces; **D06** LSP and structural-code intelligence; **D07** sessions, branching, replay, and memory; **D08** subagent and parallel-work design; **D09** worktree and execution isolation; **D10** permissions and governance; **D11** failure recovery; **D12** observability and review UX; **D13** model/provider portability; **D14** protocol and integration surfaces; **D15** evaluation methodology; **D16** licensing and code-reuse constraints.

## Cross-system architectural synthesis

The systems converge on useful harness mechanics but not on one acceptable authority model. WePLD separates five layers:

1. **Delivery Protocol:** Spec Kit's progression and Cursor/OpenCode plan UX become typed Core state transitions. Markdown, prompts, TODOs and ACP plans are import/export or presentation formats.
2. **Hermes harness:** Pi's minimal kernel and composable resources, plus typed event/tool ideas from Codex, OpenCode and OpenHands, become replaceable Agent Kernel, Skill Runtime, Hook/Extension Runtime, Context Compiler and integration/event ports. Hermes schedules only Core-admitted work.
3. **Structural intelligence:** Zed's LSP/multibuffer experience, Warp/Cursor hybrid indexing, Aider's repository graph and Claude/OpenCode LSP integration become provenance-ranked, read-only H4 observations. Exact authoritative sources outrank derived indexes.
4. **Execution operations:** Warp's control/execution/visibility split, OpenHands telemetry and Zed/Claude/Codex worktrees inform Core Delivery Control leases, Tool & Workspace/Worker Host isolation, bounded H6 scheduling and an H9 Execution Console view within Mission Control. Core owns WIP counters, capabilities, budgets and effect truth.
5. **Interop and surfaces:** ACP, MCP, JSONL/OpenAPI/SDK streams and editor review projections remain adapters around versioned Core commands, queries and events. They cannot mint capability, own truth or execute around the Effect Firewall.

This synthesis deliberately rejects a terminal-first product, cloud-first control plane, trusted-agent security boundary, prompt/Markdown authority, optional sandbox for mutating workers, unchecked plugin execution, unbounded swarms, transcript-as-ledger, summary-as-memory, and provider-count-as-portability.

## Controlled architecture spikes and ablation register

### Common experimental contract

Every experiment below is an `EvaluationRun` governed by [34_Harness_Evaluation_Protocol.md](34_Harness_Evaluation_Protocol.md). Before execution it records its treatment arms, primary metric, threshold, fixtures, repetitions, analysis, safety abort and source snapshot. Unless a row sets a stronger rule:

- use at least 30 stratified fixtures and at least three independent repetitions for nondeterministic arms, with paired randomization/counterbalancing and enough additional repetitions to meet the approved power analysis;
- hold mission, repository commit, approved specification/outcome contract, policy, tools other than the treatment, environment, budgets, maximum attempts, review rubric and allowed human intervention constant;
- retain mandatory Core policy, sandbox, Effect Firewall and evidence gates in every ordinary arm. Security-boundary tests use contained fake effects and defense-layer fault injection; they do not authorize an unsafe production baseline;
- include ambiguity, malicious repository content, prompt injection, stale data, provider failure, crash/reconnect and impossible-work fixtures where relevant;
- retain raw run manifests and negative/null results, score blind to treatment/model identity, and report distributions plus confidence/credible intervals rather than a best run;
- treat any escaped protected effect, fabricated evidence, unreconciled duplicate effect, secret leak or silent weakening of an approved contract as an automatic failure regardless of task score.

Thresholds below are proposed preregistration values. Architecture/product/security reviewers must accept them before a run; they cannot be relaxed after results are visible.

## Roadmap admission rule

Before product Core exists, reference ideas are proposed as signed, versioned repository architecture-review records named `ReferenceIdeaDecision`, with: idea ID; stated problem; first-party evidence URLs and immutable source revisions; observation/inference split; candidate target disposition; intended WePLD component; governance/threat analysis; experiment/treatment identifiers; preregistered metrics; result/evidence IDs; rollback rule; license/provenance status; decision owner/version; target milestone; and supersession history. This resolves the Baseline bootstrap without inventing a pre-H1 product domain. A later accepted ADR may import the record into an approved evaluation/provenance store without changing its evidence identity.

An idea enters an implementation plan only when all of the following are true:

1. RS-00 has produced a complete evidence/provenance record and no prohibited source reuse is involved.
2. Its spike was preregistered under document 34, executed in isolation, and met every primary safety, governance, quality and benefit threshold.
3. Negative/null results and limitations are retained; the proposed scope is no broader than the tested scope.
4. Architecture, product and security reviewers ratify or revise the matrix's candidate `Adopt`/`Adapt` target against the exact evidence. Until ratified, its admission state remains `Candidate`; `Experiment`, `Defer`, `Reject`, and unratified candidates cannot appear in an implementation Task Packet.
5. The applicable Proposed ADR is accepted, the preceding H1–H9 gate is closed, and the next milestone is explicitly authorized.
6. The implementation has an independent disable/rollback path and its ongoing regression trigger is added to H8 certification or the relevant security suite.

If an experiment fails, the idea is removed, narrowed, deferred or rejected; the roadmap is not adjusted retroactively to call the result a success.

## License, provenance, and clean-room gate

No reference-system source code, template, schema implementation, benchmark fixture or documentation example has been copied into this planning package. Architectural analysis of public behavior is not permission to reuse implementation.

Before any later reuse, WePLD must first adopt its own repository licensing/contribution policy. A separate per-component `ProvenanceReview` must then record:

- source owner, canonical URL, exact commit/tag and exact files/portions;
- upstream license text, copyright notices, patent clauses, trademark constraints, dependency licenses and whether the material is documentation, example, generated output, benchmark data or code;
- intended WePLD destination, modification/derivation method and whether a clean-room implementation is required;
- notice, attribution, source-availability, reciprocal/copyright, redistribution and network-use obligations;
- compatibility analysis against the then-approved WePLD license and distribution model;
- security/SBOM review, reviewer identities, approval version and retained evidence.

Until that policy and review exist, every implementation remains clean-room and only independently stated design principles may be considered. In particular:

- Zed and ACP are reviewed separately; protocol compatibility does not import Zed's GPL code.
- Warp's mixed MIT/AGPL boundaries prohibit assumptions about repository-wide reuse.
- Cursor and Claude Code are documentation-informed references only; proprietary source, private schemas and reverse-engineered behavior are not inputs.
- OpenHands SDK/core, enterprise areas and benchmark assets are separate provenance units.
- OpenCode Bench and any other source without an explicit reviewed license are unavailable for copying.
- Pi packages, Aider benchmark fixtures, Spec Kit extensions and all transitive dependencies carry their own provenance obligations.
